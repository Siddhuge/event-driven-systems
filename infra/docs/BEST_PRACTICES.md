# Terraform Best Practices & Standards\n\n## Code Organization\n\n### Directory Structure\n\n```\nterraform/\n├── envs/              # Environment-specific configurations\n│   ├── dev/           # Development environment\n│   └── prod/          # Production environment\n├── modules/           # Reusable infrastructure modules\n│   ├── acr/           # Azure Container Registry\n│   ├── aks/           # Azure Kubernetes Service\n│   ├── keyvault/      # Key Vault\n│   ├── network/       # Virtual Network\n│   └── resource-group/ # Resource Group\n├── docs/              # Documentation\n├── README.md          # Project overview\n├── SECURITY_CHECKLIST.md # Security pre-deployment checklist\n└── .gitignore         # Git exclusions\n```\n\n### File Naming Convention\n\n- `main.tf` - Primary resource definitions\n- `variables.tf` - Input variables\n- `outputs.tf` - Output values\n- `versions.tf` - Provider and version requirements\n- `providers.tf` - Provider configuration\n- `locals.tf` (optional) - Local variables\n\n## Code Style & Standards\n\n### Formatting\n\nAlways run `terraform fmt` before committing:\n\n```bash\n# Format single directory\nterraform fmt envs/dev\n\n# Format entire workspace recursively\nterraform fmt -recursive .\n\n# Check formatting without modifying\nterraform fmt -check -recursive .\n```\n\n### Naming Conventions\n\n#### Resource Names\n\n```hcl\n# Use snake_case for resource names\nresource \"azurerm_resource_group\" \"rg\" {  # ✓ Good\nresource \"azurerm_resource_group\" \"myRG\" {  # ✗ Bad\n\n# Use descriptive names\nresource \"azurerm_kubernetes_cluster\" \"primary\" {  # ✓ Good\nresource \"azurerm_kubernetes_cluster\" \"aks\" {  # ✓ Acceptable\n```\n\n#### Variable Names\n\n```hcl\n# Use snake_case\nvariable \"environment\" { ... }  # ✓ Good\nvariable \"env\" { ... }  # ✗ Bad (too abbreviated)\nvariable \"Env\" { ... }  # ✗ Bad (wrong case)\n```\n\n#### Local Names\n\n```hcl\nlocals {\n  common_tags        = {...}  # ✓ Descriptive\n  naming_prefix      = \"...\"  # ✓ Clear purpose\n  acl                = {...}  # ✗ Unclear\n}\n```\n\n### Variable Declarations\n\nAlways include descriptions and validation:\n\n```hcl\n# ✓ Good\nvariable \"location\" {\n  type        = string\n  description = \"Azure region for resource deployment\"\n  \n  validation {\n    condition     = can(regex(\"^[a-zA-Z\\\\s]+$\", var.location))\n    error_message = \"Location must be a valid Azure region name.\"\n  }\n}\n\n# ✗ Bad - Missing description, validation, type\nvariable \"location\" {}\n```\n\n### Output Declarations\n\nAlways include descriptions; mark sensitive data:\n\n```hcl\n# ✓ Good - Clear purpose, sensitivity marked\noutput \"key_vault_uri\" {\n  value       = azurerm_key_vault.kv.vault_uri\n  description = \"URI for accessing Key Vault secrets\"\n  sensitive   = false  # Mark as true if contains secrets\n}\n\n# ✗ Bad - No description\noutput \"kv_uri\" {\n  value = azurerm_key_vault.kv.vault_uri\n}\n```\n\n## Module Design\n\n### Single Responsibility\n\n```hcl\n# ✓ Good - Each module has single purpose\nmodule \"network\" {\n  # Manages VNet, subnets, NSGs - networking only\n}\n\nmodule \"aks\" {\n  # Manages AKS cluster - compute only\n}\n\n# ✗ Bad - Multiple concerns in one module\nmodule \"infrastructure\" {\n  # Manages networks, compute, storage, etc.\n}\n```\n\n### Input Variables\n\n```hcl\n# ✓ Good - Explicit, validated inputs\nvariable \"node_count\" {\n  type        = number\n  description = \"Number of AKS nodes\"\n  default     = 3\n  \n  validation {\n    condition     = var.node_count > 0 && var.node_count < 100\n    error_message = \"Node count must be between 1 and 99.\"\n  }\n}\n\n# ✗ Bad - Unclear, no defaults\nvariable \"nodes\" {\n  type = number\n}\n```\n\n### Outputs\n\n```hcl\n# ✓ Good - All important values exported\noutput \"cluster_id\" {\n  value       = azurerm_kubernetes_cluster.aks.id\n  description = \"Cluster resource ID\"\n}\n\noutput \"cluster_name\" {\n  value       = azurerm_kubernetes_cluster.aks.name\n  description = \"Cluster name\"\n}\n\n# ✗ Bad - Missing important values\noutput \"cluster\" {\n  value = azurerm_kubernetes_cluster.aks\n}\n```\n\n## Security Best Practices\n\n### Sensitive Data\n\n```hcl\n# ✓ Always mark sensitive outputs\noutput \"kubeconfig\" {\n  value       = azurerm_kubernetes_cluster.aks.kube_admin_config_raw\n  sensitive   = true  # Prevents printing in logs\n  description = \"Kubeconfig for admin access\"\n}\n\n# ✓ Use sensitive variables for inputs\nvariable \"admin_password\" {\n  type        = string\n  sensitive   = true  # Won't show in plan/apply output\n  description = \"Admin password\"\n}\n\n# ✗ Never put secrets in variables.tf defaults\nvariable \"password\" {\n  default = \"hardcoded_password\"  # NEVER DO THIS\n}\n```\n\n### Secrets Management\n\n```bash\n# ✓ Use environment variables for secrets (CI/CD)\nexport TF_VAR_admin_password=\"secret123\"\nterraform apply\n\n# ✓ Store in Key Vault and retrieve at runtime\ndata \"azurerm_key_vault_secret\" \"db_password\" {\n  name         = \"db-password\"\n  key_vault_id = azurerm_key_vault.kv.id\n}\n\n# ✗ Don't commit .tfvars files with secrets\ngit add terraform.tfvars  # NEVER DO THIS\n```\n\n### Network Security\n\n```hcl\n# ✓ Good - Deny by default, allow specific traffic\nresource \"azurerm_network_security_group\" \"example\" {\n  default_action = \"Deny\"\n  # Explicitly allow required ports\n}\n\n# ✗ Bad - Allow all traffic\nresource \"azurerm_network_security_group\" \"bad\" {\n  default_action = \"Allow\"\n}\n```\n\n## Testing & Validation\n\n### Pre-commit Checks\n\n```bash\n# 1. Validate syntax\nterraform validate\n\n# 2. Format check\nterraform fmt -check -recursive .\n\n# 3. Security scan (requires tfsec)\ntfsec . --minimum-severity HIGH\n\n# 4. Plan and review\nterraform plan -out=tfplan\nterraform show tfplan\n```\n\n### Local Testing\n\n```bash\n# Use dev environment for testing\ncd envs/dev\nterraform init\nterraform plan\nterraform apply\n\n# Verify resources\naz resource list --resource-group <rg-name>\nkubectl cluster-info\n```\n\n## Common Mistakes to Avoid\n\n### 1. Hard-coded Values\n\n```hcl\n# ✗ Bad\nresource \"azurerm_container_registry\" \"acr\" {\n  name = \"myreg123\"  # Hard-coded\n  location = \"eastus\"  # Hard-coded\n}\n\n# ✓ Good\nresource \"azurerm_container_registry\" \"acr\" {\n  name     = var.registry_name\n  location = var.location\n}\n```\n\n### 2. Missing Depends-On\n\n```hcl\n# ✗ Bad - May fail if dependencies not inferred\nresource \"azurerm_subnet\" \"example\" {\n  virtual_network_name = azurerm_virtual_network.vnet.name\n  # No explicit depends_on\n}\n\n# ✓ Good - Explicit dependencies\nresource \"azurerm_subnet\" \"example\" {\n  virtual_network_name = azurerm_virtual_network.vnet.name\n  \n  depends_on = [\n    azurerm_virtual_network.vnet\n  ]\n}\n```\n\n### 3. Overly Complex Modules\n\n```hcl\n# ✗ Bad - Too many concerns\nmodule \"everything\" {\n  source = \"./modules/everything\"\n  # Manages VNet, subnets, firewall, AKS, databases...\n}\n\n# ✓ Good - Focused modules\nmodule \"network\" { ... }\nmodule \"compute\" { ... }\nmodule \"storage\" { ... }\n```\n\n### 4. Operations Without Planning\n\n```bash\n# ✗ Bad - Risky\nterraform apply\n\n# ✓ Good - Always review first\nterraform plan -out=tfplan\nterraform apply tfplan\n```\n\n### 5. Committing State Files\n\n```bash\n# ✗ Never commit\ngit add terraform.tfstate\n\n# ✓ Ignore in .gitignore\nterraform.tfstate*\n*.tfvars\n```\n\n## Documentation Standards\n\n### README Structure\n\n```markdown\n# Module Name\n\n## Description\n## Security Features\n## Inputs\n## Outputs\n## Example Usage\n## Troubleshooting\n```\n\n### Inline Comments\n\n```hcl\n# ✓ Good - Explains why\nresource \"azurerm_key_vault\" \"kv\" {\n  # Prod uses premium SKU for Hardware Security Module\n  sku_name = var.environment == \"prod\" ? \"premium\" : \"standard\"\n}\n\n# ✗ Bad - Obvious from code\nresource \"azurerm_key_vault\" \"kv\" {\n  # Set the SKU name\n  sku_name = var.environment == \"prod\" ? \"premium\" : \"standard\"\n}\n```\n\n## State Management\n\n### Remote State\n\n```hcl\n# ✓ Good - Remote state with locking\nbackend \"azurerm\" {\n  resource_group_name  = \"tf-state\"\n  storage_account_name = \"tfstate\"\n  container_name       = \"tfstate\"\n  key                  = \"dev.tfstate\"\n}\n\n# ✗ Bad - Local state (no sharing, no backup)\nbackend \"local\" {\n  path = \"./terraform.tfstate\"\n}\n```\n\n### State Lock\n\n```bash\n# ✓ Good - Uses remote backend locking\nterraform apply  # Automatically locks\n\n# ✗ Bad - Local state has no locking\n# Risk: Concurrent operations corrupt state\n```\n\n## Version Control\n\n### .gitignore Essentials\n\n```gitignore\n# Local .terraform directories\n**/.terraform/*\n**/.terraform.lock.hcl\n\n# .tfstate files\n*.tfstate\n*.tfstate.*\n*.tfstate.backup\n\n# .tfvars files (contain sensitive data)\n*.tfvars\n*.tfvars.json\n!*.tfvars.example\n\n# Override files\noverride.tf\noverride.tf.json\n*_override.tf\n*_override.tf.json\n\n# IDE\n.idea/\n*.swp\n*.swo\n*~\n.vscode/*\n!.vscode/settings.json\n```\n\n### Commit Messages\n\n```bash\n# ✓ Good\ngit commit -m \"feat(aks): add Azure Policy integration for prod\"\ngit commit -m \"fix(acr): disable public access and enable quarantine\"\ngit commit -m \"docs: update security checklist\"\n\n# ✗ Bad\ngit commit -m \"update\"\ngit commit -m \"changes\"\n```\n\n## Performance Optimization\n\n### Plan Optimization\n\n```bash\n# ✓ Parallel execution (faster)\nterraform apply -parallelism=20\n\n# ✓ Skip state lock during large changes\nterraform apply -skip-lock\n\n# ✗ Serial execution (slower)\nterraform apply -parallelism=1\n```\n\n### Module Composition\n\n```hcl\n# ✓ Good - Compose modules at env level\nmodule \"network\" { ... }\nmodule \"database\" { ... }\nmodule \"aks\" { source = \"./aks\", depends_on = [module.network] }\n\n# ✗ Bad - Deeply nested modules\nmodule \"a\" {\n  source = \"./modules/a\"\n  depends_on = [module.b] # Slows down planning\n}\n```\n\n## Code Review Checklist\n\nBefore merging Terraform PRs:\n\n- [ ] Follows naming conventions\n- [ ] Includes variable descriptions and validation\n- [ ] Outputs include descriptions  \n- [ ] No hard-coded values\n- [ ] Sensitive outputs marked\n- [ ] Code formatted (`terraform fmt`)\n- [ ] Passes `terraform validate`\n- [ ] Plan reviewed and approved\n- [ ] Security checklist completed\n- [ ] Documentation updated\n- [ ] No commented-out code\n- [ ] No secrets in code/tfvars\n\n---\n\n**Last Updated:** 2024  \n**Maintained By:** Platform Team  \n**References:**\n- [Terraform Best Practices](https://www.terraform.io/docs/cloud/guides/recommended-practices.html)\n- [Azure Provider Best Practices](https://registry.terraform.io/providers/hashicorp/azurerm/latest/docs)\n
